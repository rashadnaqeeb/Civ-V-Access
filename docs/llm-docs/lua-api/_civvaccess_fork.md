# Civ V Access fork additions

Methods and `GameEvents.X` hooks added by our `CvGameCore_Expansion2.dll` fork that the auto-extracted per-class docs don't cover. The extractor scans shipped game Lua and bins by call site; these bindings have no game-side callers, so they don't appear there.

## Source of truth

`src/engine/CvGameCoreDLL_Expansion2/` -- every fork-added binding or hook has a `// CIVVACCESS:` comment block on its declaration site. Grep that marker for the canonical list and read the C++ for behavior. Additions only resolve when the deployed `CvGameCore_Expansion2.dll` is our build (default behavior of `deploy.ps1`; pass `-SkipEngine` to leave the vanilla DLL in place); on a vanilla deploy `Unit:GeneratePath` raises `luaL_error("NYI")`, the other bindings produce method-not-found errors, the `Plot:GetBestDefender` extra arg is silently ignored (binding still works at vanilla arity), and the `GameEvents.CivVAccess*` hooks just never fire.

## Current bindings

### Unit

- `unit:GeneratePath(toPlot, [iFlags=0], [bReuse=false])` -- runs the engine unit pathfinder against `toPlot`. Returns `(reachable, pathTurns)` where `pathTurns` starts at 1 (initial node) and increments at each turn boundary. Firaxis shipped this as `luaL_error("NYI")`; the fork fills in the body. `bReuse` is read with `lua_toboolean` (not `luaL_optint`) so callers can pass true / false directly.
- `unit:GetPath()` -- reads the path computed by the most recent `GeneratePath` call. Returns a 1-indexed Lua array ordered start to destination, each entry `{x, y, moves, turn, flags, revealed}` where `moves` is in `MOVE_DENOMINATOR` 60ths, `turn` matches the engine's per-node `iData2`, and `revealed` is a boolean for whether the path's tile is currently revealed to the unit's team (lets callers split a fog-crossing path into the visible prefix and the unexplored tail without a separate `Plot:IsRevealed` query). Empty array if no path is currently cached.
- `unit:ComputePath(fromPlot, toPlot, [iFlags=0])` -- runs the unit's pathfinder between two arbitrary plots without disturbing the unit's `m_kLastPath` cache. `GeneratePath` / `GetPath` always start from the unit's current position and update the cache used by mission processing; queued-waypoint computation needs to chain pathfinder runs from each prior leg's destination, which a cache-clobbering call would corrupt. Returns `(nodes, ok, legTurns)`. `nodes` matches the `GetPath` shape (1-indexed, start to destination, same per-entry fields). `legTurns` is the destination node's `iData2` (matches `GeneratePath`'s `piPathTurns`); 0 on failure -- caller treats `ok==false` as "skip this leg." Empty array on failure.
- `unit:GetMissionQueue()` -- returns the unit's pending mission queue as a 1-indexed array of `{mission, data1, data2, flags, pushTurn}`. `mission` is the `MissionTypes` enum value. Empty array if the queue is empty.
- `unit:GetBestBuildRoute(plot)` -- wraps `CvUnit::GetBestBuildRoute`. Returns `(routeId, buildId)` for the best route the unit's owner has tech for that can be built on the given plot; `(NO_ROUTE, NO_BUILD)` when the unit can't build any route there. Replaces a Lua-side reimplementation that walked `GameInfo.Builds` and re-applied tech / `Routes.Value` comparisons.

#### Vanilla fixes

- `unit:CanMoveOrAttackInto(plot, [bDeclareWar=false], [bDestination=false])` -- behavioral fix to the existing vanilla binding. Vanilla SDK calls `pkUnit->canMoveOrAttackInto(...)` and discards the return value, so the binding always returns false. The fork assigns the result so Lua callers actually get the gate's answer. Same Lua name and arity as vanilla; no caller change required.

### Plot

- `plot:GetBestDefender(eOwner, eAttackingPlayer, pAttacker, bTestAtWar, bTestPotentialEnemy, bTestCanMove, [bNoncombatAllowed=false])` -- extends the vanilla binding by one optional trailing arg. Vanilla stops at `bTestCanMove` and leaves `bNoncombatAllowed` at the C++ default of false, so Lua callers can never get civilian defenders back. Range-strike preview (where the engine accepts civilian targets) needs to read this flag, so the fork extends the arity. Existing 6-arg call sites are unaffected.
- `plot:HasLineOfSight(otherPlot, eTeam)` -- pure line-of-sight query. The vanilla `Plot:CanSeePlot` couples the `CvTargeting::CanSeeDisplacementPlot` visibility test with a range gate (`iDistance <= iRange`) and a facing gate (`shouldProcessDisplacementPlot`). Range-strike preview wants LoS in isolation; the fork passes a generous `iRange` (10000) to defeat the distance gate and `NO_DIRECTION` to short-circuit the facing gate to true. Consolidates the `LOS_PROBE_RANGE = 100` magic-number trick that the Lua side previously used and lets callers ask the visibility question they actually want.

### Game

- `Game.GetCycleUnits()` -- returns the active player's `CvUnitCycler` order as a 1-indexed Lua array of unit IDs. Same data structure the engine's `Game.CycleUnits` walks; exposing it Lua-side replaces a nearest-neighbor reimplementation and guarantees parity with the engine's own cycle order.
- `Game.GetBuildRoutePath(sx, sy, ex, ey, playerID, [route=NO_ROUTE])` -- wraps `GC.GetBuildRouteFinder().GeneratePath`. Returns a 1-indexed array ordered start to destination, each entry `{x, y}`. Empty array on no path. Callers sum build turns via `Plot:GetBuildTurnsLeft`. Flag encoding (player ID in low byte, route+1 in second byte) matches `CvBuilderTaskingAI.cpp:309-312`.

## GameEvents hooks

- `GameEvents.CivVAccessPlotRevealed(eTeam, iX, iY)` -- fires on first reveal of a plot for a team (state flip from unrevealed to revealed). Sits inside the existing `if(isRevealed(eTeam) != bNewValue)` guard in `CvPlot::setRevealed`, gated on `bNewValue == true` and `eTeam != BARBARIAN_TEAM`. Does not fire on visibility transitions of already-revealed plots -- for that, subscribe to vanilla `Events.HexFOWStateChanged`. Consumed by `CivVAccess_RevealAnnounce.lua` to drive the "tiles revealed" announcement.
- `GameEvents.CivVAccessGoodyHutReceived(playerID, eGoody, iSpecialValue)` -- fires unconditionally for the active player at the end of `CvPlayer::receiveGoody`, after the engine's MP-gated popup branch. `iSpecialValue` mirrors the popup's data2 field (gold / culture / faith amount, depending on the goody type) and is 0 for goody types that carry no scalar payload. Consumed by `CivVAccess_MultiplayerRewards.lua` and gated mod-side on `Game:IsNetworkMultiPlayer()` -- the SP popup path already announces through `CivVAccess_GoodyHutPopupAccess.lua`.
- `GameEvents.CivVAccessBarbarianCampCleared(playerID, iX, iY, iNumGold)` -- fires unconditionally for the active player when their unit clears a barbarian camp, in `CvUnit::move` after the engine's MP-gated popup branch. Consumed by `CivVAccess_MultiplayerRewards.lua` with the same MP-only gate as the goody hook.
- `GameEvents.CivVAccessForeignGoodyCleared(playerID, iX, iY)` -- sibling to `CivVAccessGoodyHutReceived`, fires when a non-active player pops a goody hut. Sits in `CvPlayer::receiveGoody` in the `else` branch opposite the active-player block. Reward type and amount are deliberately omitted (private to the actor). Consumed by `CivVAccess_ForeignClearWatch.lua`, which Lua-side filters on `pPlot:IsVisible(activeTeam)` and skips teammates.
- `GameEvents.CivVAccessForeignBarbCampCleared(playerID, iX, iY)` -- sibling to `CivVAccessBarbarianCampCleared`, fires when a non-active player clears a barbarian camp. Sits in `CvUnit::move` in the `else` branch opposite the active-player popup block. Gold amount is omitted (private to the actor's treasury). Consumed by `CivVAccess_ForeignClearWatch.lua` with the same Lua-side filters as the foreign goody hook.
- `GameEvents.CivVAccessNukeStart(attackerPlayerID, attackerUnitID, iTargetX, iTargetY, iNukeLevel, iTargetCityPlayer, iTargetCityID)` -- fires once at the top of `CvUnitCombat::ResolveNuclearCombat` before any per-entity damage is applied. Signals the mod's accumulator that a nuclear strike is starting. `iTargetCityPlayer` / `iTargetCityID` surface the target plot's city if any (sentinel -1 / -1 otherwise) so the speech can name the strike target separately from the casualty list. Skipped for meltdowns (no `pkAttacker`).
- `GameEvents.CivVAccessNukeUnitAffected(playerID, unitID, iDamage, iFinalDamage, iMaxHitPoints)` -- fires from inside `ApplyNuclearExplosionDamage` for each unit caught in the blast, BEFORE the damage is applied to the unit, so the unit handle is still valid when Lua resolves its name. The combat-member entry holds the engine's pre-computed post-damage values (delta, final, max), so the payload is complete without a post-application read. Skipped for meltdowns.
- `GameEvents.CivVAccessNukeCityAffected(playerID, cityID, iDamage, iFinalDamage, iMaxHitPoints, iPopDelta, bDestroyed)` -- fires for each city caught in the blast. Two mutually-exclusive fire sites in `ApplyNuclearExplosionDamage`: the kill branch fires BEFORE `pkCity->kill()` with `iPopDelta == 0` and `bDestroyed == 1` (full population is implicit from "destroyed"; the speech path elides a redundant pop clause); the survive branch fires AFTER the engine has applied damage and pop loss with `iPopDelta` set to the actual applied delta (capped at `prePop - 1`) and `bDestroyed == 0`. Skipped for meltdowns.
- `GameEvents.CivVAccessNukeEnd(attackerPlayerID)` -- fires at the bottom of `ResolveNuclearCombat` after damage application returns. Signals the mod to flush its accumulator and speak the composed strike result. Fires unconditionally for any `NukeStart`; the mod's `onNukeEnd` handles the no-buffer / no-affected case ("no targets hit").
- `GameEvents.CivVAccessCombatResolved(...)` -- fires from `CvUnitCombat::ResolveCombat` after the resolution dispatch returns, covering Quick + standard, units + cities on both sides, melee + ranged + air sweep. Nuclear is excluded (handled by the Nuke* hooks). Plugs the gap in vanilla's `Events.EndCombatSim`, which only fires when the gDLL combat animation completes -- `CvPreGame::quickCombat()` bypasses that, and city defenders never get an `EndCombatSim` regardless of mode. The 19-arg payload is decoded by `UnitControl.onCombatResolved`: attacker player / unit / damage-this-combat / final / max, defender player / unit / city / damage / final / max, interceptor player / unit / damage, combat kind (0 normal, 1 air-sweep into ground AA, 2 air-sweep dogfight), plot-visible-to-active-team, attacker-known, defender-known, attacker city ID. Sentinel -1 / 0 fields cover branches that don't apply (city attacker has unitID -1, unit defender has cityID -1, no-intercept pushes -1 / -1 / 0). Pre-combat damage is snapshotted earlier in `ResolveCombat` so per-combat deltas can be reported without Lua-side polling. Mod-side dedupe ensures only one spoken result per combat even if both this hook and `EndCombatSim` fire in sequence.
- `GameEvents.CivVAccessAirSweepNoTarget(attackerPlayerID, attackerUnitID)` -- fires from `CvUnitCombat::AttackAirSweep` when an air-sweep mission lands on a plot with no interceptors. The engine's own `AddMessage` for `TXT_KEY_AIR_PATROL_FOUND_NOTHING` lands in the visual notification log, which the mod has no Lua subscription for; without this hook the screen-reader path stays silent on a sweep that found nothing. Lua filters to the active player same as `CombatResolved`.
- `GameEvents.CivVAccessMissionDispatched(playerID, unitID, eMission, iData1, iData2)` -- fires from `CvDllNetMessageHandler` on the post-resolution boundary that every human-issued unit mission funnels through. Two fire sites: `ResponsePushMission` (the general mission dispatch path; covers everything that lands as a `PushMission` net message) and `ResponseSwapUnits` (the swap path that `Game.SelectionListMove` routes to when the target plot has a same-type friendly unit; dispatches as a `MOVE_TO` mission for the originating unit). In SP both dispatch near-immediately; in MP they land after the network lockstep slice processes the message. Either way, by the time the hook fires the mission has executed (move resolved, queued for next turn, or refused). Both sites fire unconditionally even when the engine refused the message or no-op'd internally (no same-type swap target found, unit destroyed between send and dispatch); the listener uses the original `iUnitID` to match against in-flight pendings and announces "action failed" rather than letting the wall-clock guess of a frame-count timeout decide.
